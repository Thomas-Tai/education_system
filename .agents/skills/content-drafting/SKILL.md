# Content Drafting Skill

> **禁止使用 Emoji**：所有生成內容（卡片標題、標籤文字、引言、總結項目等）一律不得使用 emoji。

## 觸發條件

當使用者提供以下任意組合時觸發：

- 「幫我起草一份課程內容」「生成 content.md」「根據主題寫講義」
- 明確給出主題（topic）、目標受眾（audience）、時長（duration）

若使用者只給主題而未提供受眾或時長，**主動詢問**這兩個欄位後再生成，不要自行假設。

---

## 輸入參數

| 參數 | 說明 | 範例 |
|------|------|------|
| `topic` | 課程主題（一句話描述） | "Claude Code 進階工作流程" |
| `audience` | 目標受眾，含前置知識描述 | "有 1 年以上 Claude Code 使用經驗的工程師" |
| `duration` | 課程總時長（分鐘） | "90 分鐘" |

---

## Step 1：推算章節結構

根據時長推算主章節數（`#`）與各章節分配：

| 時長 | 主章節數 | 每章節預計時間 |
|------|---------|--------------|
| 45 分鐘以下 | 2–3 個 | 12–20 分鐘 |
| 45–75 分鐘 | 3–4 個 | 15–20 分鐘 |
| 75–120 分鐘 | 4–5 個 | 18–25 分鐘 |
| 120 分鐘以上 | 5–7 個 | 20–30 分鐘 |

每個主章節內部結構：
- 1–2 個子章節（`##`）
- 每個子章節 2–4 張卡片（`### Title`）
- 適量穿插 prompt block、flow、insight box、tags

---

## Step 2：判斷受眾深度

根據 `audience` 描述判斷內容深度：

| 受眾層級 | 判斷依據 | 內容策略 |
|---------|---------|---------|
| **初學者** | 「第一次」「沒有基礎」「新手」「入門」 | 多類比、少術語、步驟詳細、每個術語首次出現時解釋 |
| **中階** | 「有一定基礎」「用過一段時間」「了解基本」 | 假設基礎概念已知，著重最佳實踐與常見誤區 |
| **進階** | 「資深」「多年經驗」「深度使用者」 | 直接跳過基礎，聚焦邊緣案例、效能、架構抉擇 |

---

## Step 3：套用「說明→範例→實作」教學節奏

**每個主章節（`#`）都必須包含三個層次**，比例可依主題調整：

### 說明層（Why / What）
- 用 1–2 張卡片（`### Title`）解釋「這是什麼」「為什麼重要」
- 用 insight box（`> **Title**`）點出核心洞察或反直覺結論
- 初學者課程：多用生活類比

### 範例層（Show）
- 用 `prompt block` 展示真實的指令、配置、程式碼片段
- 用 `[flow]` 展示操作序列或決策流程
- 至少一個具體、可執行的範例（不要用「假設有一個例子」這種模糊描述）

### 實作層（Try）
- 用帶動詞的卡片標題（「動手試試」「設定你的環境」「練習：...」）
- 提供明確的操作步驟：`[flow]` 或 `- [x]` checklist
- 結尾可以加 `[tags]` 歸納要點或常見變體

**允許例外**：純概念章節（如「課程總覽」「背景脈絡」）可以只有說明層，但需在章節引言中說明「本章節為背景鋪陳」。

---

## Step 4：元件選用指引

依據內容類型選擇對應元件：

| 內容類型 | 推薦元件 | 語法速記 |
|---------|---------|---------|
| 操作步驟、工作流程 | Flow Steps | `[flow]...[/flow]`，每行一步 |
| 兩種做法的結構化對比 | Compare | `[compare label-left="舊" label-right="新"]` + `- 左側 \| 右側` |
| 對比、分類、要點歸納 | Tags | `[tags]` + `- [green/orange/purple/blue] 文字` |
| 核心觀點、反直覺洞察 | Insight Box | `> **標題**` 後跟 `>` 段落 |
| CLI 指令、Prompt、程式碼 | Prompt Block | ` ```prompt [label="..."] ` |
| 課堂互動確認點 | Vote | `[vote id="v1" title="問題"]` + `- 選項` |
| 章節末知識檢核 | Quiz | `[quiz type="single"]` + `Q:` + `- [x] 正確` + `- [ ] 錯誤` + `Hint:` |
| 已完成/已驗證的清單 | Checklist | `- [x] 項目`（獨立行） |
| 架構圖、截圖說明（與文字並排） | Image-Text | `[image-text position="left" width="40"]` + `![alt](path)` + 文字 + `[/image-text]` |
| 章節封面 / 概念主角圖（全寬置中） | 獨立圖片 | 獨立一行 `![圖片說明](images/filename.png)`（放在 `#` 下方會成為 chapter hero） |
| 段落內輔助示意（不需 caption） | 行內圖片 | 在句子中直接寫 `![alt](path)`，渲染為 inline-image |
| 章節總結、學習成果 | Summary Grid | `[summary]` + `- **標題** \| 描述` |
| 補充知識、進階延伸 | Bonus | `[bonus title="..."]...[/bonus]` |

**圖片使用時機（不只是章節英雄）**：

- **章節英雄（chapter hero）**：每個主章節（`#`）可有一張，緊接在標題後、引言 `>` 之前，獨立一行 `![alt](path)`。不是每個章節都必須有，只在需要強烈視覺錨點時使用。
- **內文獨立圖片**：在 `##` 或 `###` 段落間插入，用於展示架構圖、截圖、示意圖、操作結果。獨立一行 `![alt](path)`，build 會自動渲染為置中的 `<figure>` + caption。
- **圖文並排 `[image-text]`**：當圖片是某段文字的輔助（例如 UI 截圖 + 欄位說明），使用 image-text 元件讓兩者並列，避免上下堆疊造成視線斷裂。
- **行內圖片**：僅在圖片很小、屬於語句一部分時使用（例如 icon、小型示意）。多數教學場景應改用獨立圖片或 image-text。

**元件使用節奏**（一個主章節的建議組合）：
- 說明層：卡片（`###`）+ Insight Box；概念性主題可加一張**獨立圖片**或 **image-text** 輔助理解
- 範例層：Prompt Block 或 Flow Steps；若有兩種做法對比 → Compare；有 UI 或輸出結果 → **image-text** 圖文並排
- 實作層：Flow Steps 或 Checklist；互動課程可加 Vote 或 Quiz；操作練習可搭配**獨立圖片**顯示預期結果

**使用時機**（互動元件）：
- `[compare]`：兩種方案有結構化的逐點對比（非泛泛而談）
- `[vote]`：適合放在章節開頭暖身（預測問題）或章節末確認理解
- `[quiz]`：知識密集章節的末尾，驗收單一概念

**避免**：
- 裸段落（loose text）— 段落式文字應改為卡片 + 條列
- 過度使用 insight box（一個主章節最多 2–3 個）
- 連續多張卡片無任何 flow 或 prompt block（顯得單調）
- 每個章節都加 vote/quiz（互動元件應選擇性使用，每門課 3–5 個為宜）

---

## Step 5：生成 content.md

生成時遵守以下規則：

1. **第一個章節前不加 `---`**，從第二個章節起每個 `#` 前加 `---`
2. **所有 `#` 標題使用全形冒號**：`# LABEL：TITLE`（如 `# 環境準備：安裝與初始設定`）
3. **引言（lead）緊接在 `#` 後**，用 `>` 開頭，一句話點出本章節的核心價值
4. **最後一個章節固定為總結**，使用 `[summary]` 歸納各章節的學習成果
5. 每張卡片的條列重點應為**真實內容**，而非 `<!-- TODO -->` 佔位符
6. Prompt block 的程式碼應為**真實可用的指令或範例**，不使用虛構語法
7. **圖片規則**：
   - **章節英雄**：若想為某主章節加上 chapter hero，將獨立一行的 `![alt](images/xxx.png)` 放在 `#` 標題與 `>` 引言之間（中間空一行）。每門課程 2–4 張 hero 為宜，不要每章都加。
   - **內文獨立圖片**：在段落之間獨立成行，前後各空一行。`alt` 必填且為有意義的描述（會自動成為 figcaption）。路徑使用**相對路徑**：`images/xxx.png`（對應課程目錄下的 `assets/images/xxx.png`，build 會自動解析）。
   - **圖文並排**：使用 `[image-text position="left|right" width="N"]...[/image-text]`，圖片與文字都要有實際內容。
   - **無圖時佔位**：若該位置未來應加圖但目前無圖，保留 `<!-- TODO: 建議加圖 — 說明用途 -->` 供 `image-generator` Skill 或後續作者定位。

---

## Step 6：更新 config.yaml（如果存在）

生成 content.md 後，檢查課程目錄是否有 `config.yaml`：

- **若無**：在回覆中提醒使用者參考 `course-page-generator` Skill 的 Step 2 建立 config
- **若有**：根據草稿內容建議更新 `quotes.opening` 和 `quotes.closing`，提供具體文字但由使用者確認後再寫入

---

## 輸出格式

回覆使用者時包含：

1. **章節大綱**（一張表，列出章節編號、標題、預計時間、使用的主要元件）
2. **完整 content.md 內容**（程式碼區塊包裹，方便複製）
3. **缺圖位置清單**（若有 `<!-- TODO: 建議加圖 -->` 的地方，彙整列出）
4. **下一步建議**（提示可執行 build 或先用內容審閱 Skill 檢查教學節奏）
