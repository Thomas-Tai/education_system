# Content Review Skill

> **禁止使用 Emoji**：審閱報告中的所有標題、建議文字一律不得使用 emoji。

## 觸發條件

當使用者說：

- 「審閱課程內容」「幫我看 content.md」「內容審閱」
- 「分析教學節奏」「章節結構好嗎」「哪裡可以改進」

---

## 輸入

- **必要**：`content.md` 的路徑（或貼上內容）
- **選填**：目標時長（分鐘），用於估算時間是否超出

若使用者未指定 content.md 路徑，先用 Glob 工具在當前工作目錄搜尋 `content.md`，找到後告知使用者確認再進行審閱。

---

## Step 1：解析結構

解析 content.md，建立章節索引：

```
主章節列表：
  1. # LABEL：TITLE
     - ## 子章節 A
       - ### 卡片 1, 2, 3...
       - [flow] / prompt block / insight / tags...
     - ## 子章節 B
       - ...
  2. # LABEL：TITLE
     - ...
```

計算每個主章節的元件清單（類型 + 數量）。

---

## Step 2：節奏分析（每個主章節）

對每個主章節（`#`），依序判斷三個教學層次是否存在：

### 說明層（Why / What）

存在條件（滿足任一即可）：
- 有 `### 卡片` 包含解釋性文字（列出概念、定義、比較）
- 有 insight box（`> **Title**`）

缺失特徵：
- 章節直接跳入操作步驟，沒有解釋「為什麼這樣做」
- 所有卡片都是動詞開頭的操作指令

### 範例層（Show）

存在條件（滿足任一即可）：
- 有 prompt block（` ```prompt `）
- 有具體的 `[flow]` 展示操作流程
- 有 insight box 搭配具體案例

缺失特徵：
- 只有文字描述，沒有任何可執行的範例或指令
- `[flow]` 的步驟過於抽象（如「步驟一：開始」「步驟二：完成」）

### 實作層（Try）

存在條件（滿足任一即可）：
- 有帶動詞的卡片標題（「動手」「練習」「設定」「嘗試」等）
- 有可執行的 `[flow]` 步驟（具體指令或可操作動作）
- 有 `- [x]` checklist 代表已驗證的操作清單

缺失特徵：
- 章節結束於理論說明，沒有引導學員動手
- 有 prompt block 但缺乏「現在你來試試」的引導

---

## Step 3：全篇品質檢查

### 3-1 裸段落偵測

掃描 content.md，找出不在任何結構化元件內的純文字段落（loose text）：

- 不在 `###` 卡片內的段落 → 建議移入卡片或 insight box
- 緊跟 `#`/`##` 的段落（非 `>` 引言）→ 建議改為 `>` 引言或卡片

### 3-2 卡片密度

統計每個主章節的卡片數量，給出密度評估：

| 卡片數 / 主章節 | 評估 |
|--------------|------|
| 1–2 張 | 過少，內容可能太稀薄，或應該移至其他章節合併 |
| 3–6 張 | 合適範圍 |
| 7–10 張 | 偏多，考慮拆出子章節（`##`）分組 |
| 11 張以上 | 建議拆為兩個主章節 |

### 3-3 元件多樣性

檢查全篇是否過度依賴單一元件：

- 全篇沒有 `[flow]` → 建議加入流程說明
- 全篇沒有 prompt block → 確認是否為純概念課程（合理）或缺乏具體示範
- 全篇沒有 insight box → 建議找出 2–3 個值得深化的核心觀點
- 全篇沒有 `[summary]` → 缺少總結，建議在最後章節加入

### 3-4 時間估算

依以下基準估算全篇總時長：

| 元件 | 預估講解時間 |
|------|------------|
| `### 卡片`（含 3–5 條列） | 2–3 分鐘 |
| `### 卡片`（含 6+ 條列） | 4–5 分鐘 |
| `[flow]`（3–5 步驟） | 3–4 分鐘 |
| Prompt block | 2–3 分鐘（含示範） |
| Insight box | 1–2 分鐘 |
| `[tags]` | 1 分鐘 |
| `[youtube]` | 影片長度 + 1 分鐘說明 |
| `[summary]` | 2–3 分鐘 |

輸出：「預估總時長：XX–YY 分鐘」，若有目標時長則比較差距。

### 3-5 圖片多樣性與覆蓋

統計以下四類圖片的分布與密度：

| 類型 | 語法 | 建議密度 |
|------|------|---------|
| **chapter hero** | `#` 下方、`>` 之前的 `![alt](path)` | 每門課 2–4 張，不宜每章都有 |
| **內文獨立圖片** | 段落間的 `![alt](path)` | 每個主章節平均 1–3 張 |
| **image-text** | `[image-text]...[/image-text]` | 有 UI / 截圖輔助時使用，每門課 3–8 個 |
| **行內圖片** | 句子中的 `![alt](path)` | 僅 icon / 小示意，少用 |

檢查重點：

- **hero 過度**：主章節 hero 超過 5 張，或每章都有 → 建議挑選 2–4 個最具視覺錨點的章節保留，其他改為內文獨立圖片
- **hero 缺失**：超過 5 個主章節卻零張 hero → 建議在「開場」或「總結」加入 1–2 張
- **長章節缺圖**：某主章節超過 8 張卡片、3 個子章節，卻沒有任何圖片 → 建議加入獨立圖片或 image-text 緩解閱讀疲勞
- **TODO 殘留**：仍有 `<!-- TODO: 建議加圖 -->` 或 `[image-here]` 未處理 → 建議執行 `image-generator` Skill 批次生成
- **過度依賴 image-text**：每個子章節都用 image-text → 視覺單調，建議改用獨立圖片或穿插 flow / prompt block
- **圖片 alt 空泛**：如 `![image](...)`、`![img](...)`、`![](path)` → 建議補上有意義的描述（會成為 figcaption）

## Step 4：輸出審閱報告

### 存檔判斷

| 情境 | 行為 |
|------|------|
| 由 `topic-to-page` Skill 呼叫 | 直接將報告寫入 `<course_dir>/review-report.md`，不詢問 |
| 使用者說「審閱並存檔」「存成報告」「save report」 | 直接寫入 `<course_dir>/review-report.md` |
| 一般單獨觸發 | 在對話中輸出完整報告，並詢問「是否要將報告存為 review-report.md？」；確認後再寫入 |

寫入時使用 Write 工具，路徑為 `<course_dir>/review-report.md`（覆蓋舊檔）。

### 報告結構如下：

```markdown
## 章節節奏診斷

| 章節 | 說明 | 範例 | 實作 | 整體評估 |
|------|------|------|------|---------|
| # 章節一 | ✅ | ⚠️ 缺具體指令 | ❌ | 中 |
| # 章節二 | ✅ | ✅ | ✅ | 佳 |
| # 章節三 | ✅ | ❌ | ❌ | 需改善 |

（圖例：✅ 完整  ⚠️ 薄弱  ❌ 缺失）

---

## 各章節改善建議

### 章節一：TITLE
- 範例層：缺少具體指令，建議在「## 子章節 A」後加入一個 prompt block
  ```
  （示意用的 prompt block 範例）
  ```
- 實作層：建議在章節末新增一張卡片「動手試試：...」，引導學員實際操作

### 章節三：TITLE
- 範例層：`[flow]` 的步驟描述過於抽象（如「步驟二：執行」），建議補充具體操作
- 實作層：整個章節缺少引導動手的段落，建議在最後的卡片加入可操作的 checklist

---

## 全篇品質

**裸段落**：第 45 行的段落「...」不在任何元件內，建議移入卡片或改為 insight box

**卡片密度**：章節三有 12 張卡片，建議拆為兩個主章節

**元件多樣性**：全篇缺少 `[summary]`，建議在最後章節加入學習成果總結

**圖片多樣性**：
- chapter hero：2 張（章節一、章節三），密度合理
- 內文獨立圖片：4 張，分布均勻
- image-text：1 個（章節四 UI 說明），建議章節二的「比較舊版與新版」也加一個
- 行內圖片：0 個（正常）
- 殘留 TODO：第 78 行 `<!-- TODO: 建議加圖 -->` 未處理，建議執行 image-generator Skill

**時間估算**：預估總時長 75–90 分鐘（目標 90 分鐘，時間充裕）

---

## 優先修改清單

1. 【高優先】章節三補充範例層：加入 prompt block 展示具體操作
2. 【高優先】全篇補充 [summary]
3. 【中優先】章節一補充實作層：加入引導動手的卡片
4. 【低優先】章節三考慮拆分為兩個主章節
```

---

## 輸出原則

- **具體而非模糊**：每條建議都指出具體位置（哪個子章節、哪張卡片之後）
- **提供示意**：對「缺少範例」「缺少 flow」等問題，附上一小段示意 Markdown
- **不重寫原稿**：審閱 Skill 只輸出建議，不直接生成修改後的 content.md；若使用者要求修改，切換到 `content-drafting` 或直接編輯
- **正面肯定**：先指出做得好的地方，再給出改善建議
